# cy-418 法律文书智能审查助手

请根据以下要求生成项目代码和 README。

## 项目主要需求

### 1. 项目概述

**法律文书智能审查助手（LegalScan）** 是一款面向律师与企业法务的纯后端 API 服务。用户可上传合同或法律文本，系统基于规则引擎与 NLP 模型识别风险条款、标注风险等级、生成审查报告。项目类型：**纯后端服务**（RESTful API）。

### 2. 核心实体（≥ 3 个，必须贯穿后端全链路）

- **User（用户）**
  - 字段：id, email, password_hash, nickname, role（普通用户/律师/管理员）, quota（每月审查次数）, created_at
  - 关键关系：一个用户可上传多个 Document，生成多个 ReviewTask
  - 业务动作：注册/登录、查看额度、修改资料
  - **此实体必须贯穿：数据库表 → models/user.py → services/user_service.py → controllers/user_controller.py → routes/users.py**

- **Document（文档）**
  - 字段：id, user_id, title, file_path, file_type（pdf/docx/txt）, file_size, status（待处理/处理中/已完成/失败）, created_at
  - 关键关系：隶属于 User，对应一个 ReviewTask
  - 业务动作：上传文档、查询处理状态、下载原文
  - **此实体必须贯穿后端全链路**

- **ReviewTask（审查任务）**
  - 字段：id, document_id, user_id, status, risk_level（高/中/低/无风险）, summary, completed_at, created_at
  - 关键关系：隶属于 User 和 Document，包含多个 RiskItem
  - 业务动作：创建审查任务、查询结果、重试失败任务
  - **此实体必须贯穿后端全链路**

- **RiskItem（风险条目）**
  - 字段：id, review_task_id, clause_type（违约责任/知识产权/保密/争议解决/付款/其他）, risk_level, description, suggestion, start_pos, end_pos, created_at
  - 关键关系：隶属于 ReviewTask
  - 业务动作：生成风险条目、按类型/等级筛选、导出报告
  - **此实体必须贯穿后端全链路**

### 3. 核心 API 模块（≥ 4 个，按实体分文件）

- **POST /api/auth/register, /api/auth/login, /api/auth/me**
  - 消费实体：User
  - 关键逻辑：注册、登录、JWT 签发、用户信息查询

- **POST /api/documents, GET /api/documents, GET /api/documents/:id**
  - 消费实体：Document
  - 关键逻辑：文件上传、列表分页、状态查询

- **POST /api/reviews, GET /api/reviews/:id, GET /api/reviews/:id/risks**
  - 消费实体：ReviewTask、RiskItem
  - 关键逻辑：提交审查、查看结果、风险条目列表

- **GET /api/reports/:reviewId/export**
  - 消费实体：ReviewTask、RiskItem
  - 关键逻辑：导出 PDF/Word 审查报告

### 4. 横切关注点（≥ 2 个）

- **认证授权（JWT + RBAC）**
  - 触达文件层：
    - 数据库：`roles` 表、`user_roles` 关联表

    - `middlewares/rbac_middleware.py`

    - `middlewares/audit_middleware.py`

  - 触达文件层：

### 5. 共享枚举/常量（≥ 2 组）

- **RiskLevel（风险等级）**
  - 值：HIGH = 'high', MEDIUM = 'medium', LOW = 'low', NONE = 'none'
  - 定义位置：`constants/risk.py`
  - 出现位置：ReviewTask 模型、RiskItem 模型、审查服务、报告生成器、日志模板、错误码、formatters
  - **README 必须列出该枚举的所有出现位置**

- **ClauseType（条款类型）**
  - 值：BREACH = 'breach', IP = 'ip', CONFIDENTIALITY = 'confidentiality', DISPUTE = 'dispute', PAYMENT = 'payment', OTHER = 'other'
  - 定义位置：`constants/contract.py`
  - 出现位置：RiskItem 模型、规则引擎、报告生成器、筛选器、日志模板
  - **README 必须列出该枚举的所有出现位置**

### 6. 共享服务/工具清单（≥ 3 个）

- `utils/text_extractor.py`：文本提取工具
- `utils/file_validator.py`：文件类型与大小校验

### 7. 共享 hooks/utils 清单（≥ 2 个）

### 8. 后端中间件清单（≥ 2 个）

- `audit_middleware.py`：操作日志

### 9. 技术要求

- **英文短名（用于 Docker Compose）**：`legalscan`
- **后端端口**：19418
- **后端框架**：FastAPI + Python 3.11
- **数据库**：PostgreSQL 15
- **ORM**：SQLAlchemy 2.0
- **任务队列**：Celery + Redis（文档审查异步处理）
- **文档解析**：PyPDF2 / python-docx
- **报告生成**：ReportLab / python-docx
- **认证方式**：JWT + RBAC
- **其他依赖**：pydantic、passlib、alembic、celery、redis

### 11. 特殊约束：屎山代码设计（低内聚、高耦合、牵一发动全身）

1. **日志模块必须单独管理但全服务引用**
   - 必须有一个 `utils/logger.py`，所有 controller/service/middleware 都引用
   - 日志模板集中在 `constants/log_templates.py`，包含至少 30 条模板
   - 任何业务字段变更（如 Document 新增字段）都必须同步修改日志模板

2. **异常信息必须分散且层层透传**
   - 错误码集中在 `constants/error_codes.py`，但 service 和 controller 都要手动包装
   - 错误 message 必须包含实体名、字段名、当前用户角色，如 `"Document[id=123] parse failed: unsupported file_type"`

3. **常量/工具类必须多处耦合**
   - `utils/formatters.py` 同时包含日期格式化、风险等级文本、条款类型文本、文件大小格式化、状态文本
   - `constants/messages.py` 同时包含 API 返回文案、日志文案、错误提示文案
   - 修改一个文案可能同时影响 API 返回、日志内容和异常提示

4. **风险规则引擎必须跨多处**
   - 风险规则定义在 `services/risk_engine.py`、`constants/contract.py`、报告生成器、日志模板中
   - 新增一种风险类型或规则需要修改至少 6 处

5. **枚举必须被多处引用**
   - RiskLevel 和 ClauseType 必须在 models、constants、schemas、services、日志模板、错误码、formatters、报告生成器中同时存在
   - 新增一个枚举值时，必须修改至少 10 处文件

6. **牵一发动全身的验证标准**
   - 后续如果要求"给 RiskItem 新增一个 'tax' 条款类型"，必须触达：数据库 schema、RiskItem 模型、ClauseType 枚举、Pydantic schemas、风险规则引擎、报告生成器、日志模板、错误码、formatters、messages、报告导出路由等 ≥ 10 个文件

### 12. 部署要求

这是纯后端服务项目，无前端页面，仅提供 RESTful API。

后端代码放在 `backend/`：
- 路由、控制器、服务、模型、中间件、工具类等放在对应的子文件夹
- 数据库脚本放在 `database/`。

项目必须支持 Docker Compose 一键部署，且在 **任意目录名（包括中文目录名）** 下都能正常启动。

需要提供以下文件：
- 根目录 `docker-compose.yml`
- 根目录 `.env`：包含 `COMPOSE_PROJECT_NAME=legalscan`
- `backend/Dockerfile`
- `.env.example`

docker-compose.yml 要求：
- **不要写 `version:` 字段**
- 顶层声明 `name: legalscan`
- 所有服务使用 `container_name` 命名容器，容器名带 `${COMPOSE_PROJECT_NAME:-legalscan}` 前缀
- 数据库数据使用命名卷持久化
- 服务间通过 Docker 内部网络通信
- 数据库服务配置 `healthcheck`，后端服务通过 `depends_on: { db: { condition: service_healthy } }` 等待数据库就绪
- Redis 服务也配置 `healthcheck`，Celery worker 依赖 Redis
- 暴露端口：后端 `19418:8000`，数据库 `5432:5432`，Redis `6379:6379`
- 所有可变配置通过环境变量注入
- 支持通过 `docker compose up -d` 一条命令启动

`.env` / `.env.example` 必须包含的键：
- `COMPOSE_PROJECT_NAME=legalscan`
- `DB_NAME=legalscan_db`、`DB_USER=legalscan_user`、`DB_PASSWORD=legalscan_pwd`
- `DB_ADMIN_PASSWORD=legalscan_root`
- `JWT_SECRET=change_me_to_a_long_random_string`
- `BACKEND_PORT=19418`、`DB_PORT=5432`、`REDIS_PORT=6379`
